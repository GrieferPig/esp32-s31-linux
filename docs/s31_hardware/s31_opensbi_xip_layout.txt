ESP32-S31 OpenSBI Flash XIP layout

Goal:
- Keep OpenSBI text/rodata in flash and execute through the flash MMU window.
- Keep OpenSBI writable data, BSS, scratch, stack, and heap in reserved cached
  PSRAM so the core 0 factory app can keep HP SRAM.
- Keep the OpenSBI partition size fixed at 512KB.
- Boot OpenSBI/Linux on HP core 1. Core 0 boots the ESP-IDF factory app in
  single-core mode and then parks after releasing core 1.

Address contract:
- The factory app maps the opensbi partition for instruction access at 0x40030000.
- OpenSBI must be linked with FW_TEXT_START=0x40030000.
- OpenSBI writable sections are linked with FW_RW_START=0x50F00000.
- The top 1MB of the 16MB cached PSRAM window is reserved for OpenSBI RW:
  0x50F00000..0x51000000.
- Linux receives only the lower 15MB of cached PSRAM in its DT memory node:
  0x50000000..0x50F00000.
- Linux xipImage is mapped at 0x400B0000.

Partition image format:
- Offset 0: raw OpenSBI fw_jump.bin.
- Next byte after fw_jump.bin: OpenSBI FDT blob.
- Padding: zero bytes up to partition_size - 4.
- Last 4 bytes of the 512KB partition: little-endian uint32 file offset of the FDT blob.

Boot flow:
- The ESP-IDF second-stage bootloader uses the normal IDF partition selection
  and boots the factory app on core 0.
- The factory app is built with CONFIG_FREERTOS_UNICORE=y and
  CONFIG_ESP_SYSTEM_SINGLE_CORE_MODE=y, so IDF does not start core 1.
- The factory app verifies PSRAM, maps OpenSBI/Linux with esp_mmu_map_virt(),
  copies initramfs to 0xC0800000, configures permissive APM, and writes the
  core 1 boot address.
- Core 1 starts in a small IRAM trampoline. The trampoline clears interrupt
  state, installs a full-space PMP entry for hart 1, and jumps to OpenSBI with:
  - a0 = 1
  - a1 = fdt_addr
- Core 0 parks after releasing core 1 to avoid IDF timer/interrupt activity
  racing OpenSBI/Linux during bring-up.

OpenSBI requirements:
- fw_jump.S must report boot hart 1.
- fw_base.S must allow hart 1 through the early entry check.
- The OpenSBI/Linux FDT CPU node must describe cpu@1/reg=<1> for the single
  Linux hart.

Build-time checks:
- The generated fw_payload.bin must be exactly 512KB.
- OpenSBI + FDT must fit in the first 512KB - 4 bytes.
- Link-time assertions require FW_TEXT_START=0x40030000 and FW_RW_START in the
  reserved PSRAM range.

OpenSBI XIP relocation rule:
- fw_base.S copies .data from flash LMA to FW_RW_START before the boot lottery.
- The R_RISCV_RELATIVE relocation loop then patches pointers inside .data, for
  example platform.ops at platform + 0x60.
- Do not copy .data a second time after relocation. Doing so restores the raw
  flash image, zeros relocated pointers, and can fault on a null load in
  sbi_platform_ops().

Early fault note:
- OpenSBI's first instructions copy .data from flash LMA to FW_RW_START.
- A store/AMO access fault before the OpenSBI banner usually points at handoff
  protection state, PSRAM mapping, or PMP/APM state, not the FDT offset.

ESP32-S31 S-mode CLIC interrupt findings
Date: 2026-06-27
Cleaned/updated: 2026-07-22
Ground truth: docs/s_mode_clic_interrupt.md

Scope
-----
This file keeps only the current, non-obsolete findings for configuring
ESP32-S31 CLIC interrupts to S-mode.  Older negative tests that were caused by
missing CLIC multi-privilege configuration, PASS_LEVEL-only routing attempts,
or S/U CLIC-base mirror confusion have been removed.

Current model
-------------
- ESP32-S31 HP core is a pure CLIC-mode core.  mtvec.MODE and stvec.MODE read
  back as CLIC mode 0x3.
- The standard RISC-V interrupt path is not available for this bring-up:
  mie/mip/mideleg/uie/uip are unavailable or hardwired, and CSR sie (0x104)
  is illegal from both M-mode and S-mode.
- mideleg is hardwired to 0.  In CLIC mode, interrupt target privilege is
  selected by each implemented CLIC slot's clicintattr[i].MODE field after
  mcliccfg.NMBITS has been configured.
- mcause contains implementation status bits in the upper bits on traps.
  Always parse the cause/interrupt ID with:
      cause_code = mcause & 0xfff
- S-mode can clear IP bits directly using the S-mode CLIC base address.
  On S31, the non-standard S-mode CLIC base is available at CSR 0x150 (`sclicbase`),
  which evaluates to `0x10A00000`. S-mode must write to `sclicbase + 0x1000 + 4*i`
  to clear the pending bit. CPU_APM must be configured by M-mode to permit S-mode
  peripheral access.

CLIC bases and IDs
------------------
- M CLIC base:
      mclicbase CSR 0x350 = 0x10800000
- Per-interrupt register block:
      0x10801000 + 4 * raw_clic_id
      byte +0: clicintip
      byte +1: clicintie
      byte +2: clicintattr
      byte +3: clicintctl
- External raw CLIC IDs are 16..47.  When writing interrupt matrix SOURCE_MAP
  registers directly, write the raw CLIC ID, not an ESP-IDF allocator interrupt
  number.
- Implemented CLIC slots observed in scans:
      local/system: 1, 3, 5, 7, 11
      external:     16..47
  Other raw IDs in 0..47 read as unimplemented/reserved in the CLIC attr scan.

Required S-mode CLIC configuration
----------------------------------
1. Enable multi-privilege CLIC mode in mcliccfg.

   The relevant field is NMBITS[6:5].  Configure effective NMBITS=1 before
   relying on clicintattr.MODE for M/S/U routing.

   Example:
       volatile uint32_t *mcliccfg = (volatile uint32_t *)0x10800000;
       *mcliccfg = (*mcliccfg & ~(3U << 5)) | (1U << 5);

2. Configure the target CLIC slot through the M CLIC per-interrupt word.

   Example for raw CLIC ID 1, commonly used as Supervisor Software Interrupt:

       volatile uint32_t *clicint_1 =
           (volatile uint32_t *)(0x10801000 + 4 * 1);

       /*
        * Write ATTR before IP to ensure hardware recognizes edge-triggering!
        * byte 2: clicintattr = 0x42   MODE=1 (S), TRIG=1, SHV=0
        * byte 3: clicintctl  = 0xff   highest level/priority
        * byte 1: clicintie   = 0x01   enable
        * byte 0: clicintip   = 0x01   trigger pending
        */
       volatile uint8_t *c = (volatile uint8_t *)clicint_1;
       c[2] = 0x42;
       c[3] = 0xff;
       c[1] = 0x01;
       c[0] = 0x01;

   ESP-IDF privilege encodings used for MODE:
       PRV_U = 0
       PRV_S = 1
       PRV_M = 3

3. Configure stvec for S-mode CLIC trap entry.

       csrw stvec, ((handler_base & ~0x3f) | 3)

   The base must be 64-byte aligned.  stvec.MODE reads back as 3.

4. Enable S-mode interrupt state before mret to S-mode.

       mstatus.SIE  = 1
       mstatus.SPIE = 1

   Do not use CSR sie; it is illegal on S31.

5. If the interrupt is an external peripheral source, connect that peripheral
   source to the selected raw CLIC ID through the interrupt matrix SOURCE_MAP.
   The CLIC slot's MODE controls the delivery privilege.  Do not treat
   interrupt matrix PASS_LEVEL alone as sufficient S-mode delivery.

6. In the S-mode handler, read sclicbase to clear the pending bit directly.

       static void __attribute__((interrupt("supervisor"), aligned(64)))
       test_s_clic_handler(void)
       {
           uint32_t sclicbase;
           __asm__ volatile ("csrr %0, 0x150" : "=r"(sclicbase));
           
           volatile uint8_t *ip = (volatile uint8_t *)(sclicbase + 0x1000 + 4 * 1);
           *ip = 0;          /* clear clicintip for the relevant ID */
       }

7. PMP and APM Requirements for S-mode:
   - M-mode must configure PMP (e.g. pmpaddr0=-1, pmpcfg0=0x1F) to allow S-mode
     instruction fetch and data access.
   - M-mode must configure CPU_APM (Access Permission Monitor) to allow S-mode
     (REE_MODE 1) to access the peripheral memory space, otherwise accessing
     sclicbase will cause a Store/Load access fault.
   - mintthresh (0x347) and sintthresh (0x147) must be cleared to 0.

8. Triggering Interrupts via Software:
   - Only edge-triggered interrupts (ATTR bit 1 = 1) can have their IP bit written
     by software.
   - You must write the ATTR register byte FIRST to configure it as edge-triggered,
     BEFORE attempting to write 1 to the IP byte. Writing them simultaneously via
     a 32-bit word write will fail to latch the IP bit.

Confirmed useful observations
-----------------------------
- mcliccfg WARL writeback can expose effective NMBITS=1:
      writing 0x00000063 while preserving existing low fields read back
      0x00000023 in the observed scan.
- clicintattr.MODE is writable on implemented M CLIC slots after NMBITS is
  configured.  Observed readbacks accepted MODE=0, MODE=1, and MODE=3.
- clicintctl has 3 effective high bits on the tested S31.  The low 5 bits are
  hardware-forced to 1.  For example:
      write 0x2a -> read 0x3f
  This is normal CLIC control-bit WARL behavior and is not a privilege selector.
- stvec, sscratch, sepc, scause, stval, sstatus, and satp are usable S-mode
  trap/MMU CSRs for OpenSBI-to-S-mode handoff.  stvec mode bits are forced
  to 3 with a writable, 64-byte-aligned base.

Linux/OpenSBI CLIC return and timer forwarding (2026-07-22)
------------------------------------------------------------
- ESP-IDF is explicit about the architectural requirement.  In
  components/riscv/vectors.S, the CLIC interrupt return path restores the
  saved mcause before mstatus and mret because mcause contains the former CPU
  priority that xret copies back into mintstatus.
- A temporary bootloader S-mode probe confirmed that the same rule applies to
  scause on S31.  The measured sequence was:
      phase=3 outer=98000021 synthetic=98ff0021
      restored=98000021 after_sret=88000021 second=98000021
  scause is writable, the complete saved value reads back correctly, and sret
  consumes the saved cause/priority state.  The probe was removed after the
  result was captured.
- Linux therefore restores the complete saved scause, then sstatus and sepc,
  and executes exactly one sret for each hardware interrupt.  A temporary
  design that used an early synthetic sret plus the final sret was invalid:
  it popped the CLIC return state twice.
- OpenSBI applies the corresponding mcause-before-mstatus restore order to its
  M-mode trap return.

Timer forwarding race and final fix
-----------------------------------
- OpenSBI receives hardware M-timer ID7 and forwards it by setting S-timer ID5
  pending.  If it injects ID5 while an S-mode CLIC level is still active, S31
  can accept the pending timer on the same boundary as the outer sret and save
  SPIL=0xff.  The failure then blocks all later S interrupts.
- The useful stuck signature is a cause ending in 0x00ff0005 (observed as
  0x98ff0005 in the Linux trace and 0x88ff0005 in a JTAG CSR snapshot), with
  sintstatus.SIL=0xff and completed GDMA/UART/timer slots still IP=1 and IE=1.
  Normal non-nested entries have SPIL=0, for example 0x98000005.
- The final fix is in the OpenSBI forwarding source, not the AXI GDMA driver.
  Before injecting ID5, OpenSBI reads sintstatus.  If SIL is non-zero, it
  rearms the M timer for a 100 us retry; it injects ID5 only after the S-mode
  CLIC level has been unwound.  The restart-list dequeue must use restart_list,
  not event_list.
- Forwarded S-timer ID5 uses clicintctl=0x3f, matching the default Linux
  external interrupt level.  Driver-level timer masking and deferred CLIC
  unmasking are not required and are not present in the AXI GDMA driver.

AXI GDMA validation
-------------------
- All three AXI GDMA channel pairs are exposed through raw CLIC IDs 33, 34 and
  35, backed by interrupt-matrix sources 48, 49 and 50.
- dmatest configuration: one thread per channel, three channels, 128 KiB test
  buffers, fixed 64 KiB copies, norandom=Y.
- Stress result: 1000 copies per channel, 0 failures on all channels, about
  19.1 iops and 1222-1225 KiB/s per channel.  IRQ counts increased exactly by
  1000 for each channel.
- Final trace-free image regression: 200 copies per channel, 0 failures,
  19.64-19.73 iops and 1257-1263 KiB/s; IRQ counts were exactly 200/200/200.

AXI GDMA dmaengine expansion and performance (2026-07-22)
----------------------------------------------------------
- A 3-channel x 200-copy comparison used 64 KiB fixed transfers.  A 32-byte
  AXI burst averaged about 19.75 iops while 128-byte burst averaged about
  19.88 iops, both with zero failures.  The driver therefore uses 128-byte
  bursts.  The small difference is expected because this dmatest setup is
  dominated by non-coherent cache synchronization.
- TX interrupt-matrix sources 51, 52 and 53 are routed to raw CLIC IDs 36, 37
  and 38.  RX remains on source 48-50 / raw ID 33-35.
- The dmaengine implementation now supports memcpy, memset, slave SG, cyclic,
  residue reporting, TX descriptor-error IRQ, hardware LINK_RESTART append,
  contiguous allocations from the internal descriptor pool, and IDF-style
  64-byte RX head/tail stash recovery through the uncached PSRAM alias.
- Random source/destination offsets exercised the stash path: 200 memcpy tests
  per channel and 200 memset tests per channel completed with zero failures.
  Four threads per channel (1200 total memcpy operations) completed with zero
  failures and exercised multiple outstanding descriptors/hardware append.
- A userspace cached-memory memcpy benchmark moved 128 MiB per case and measured
  51.78 MiB/s for 64 KiB blocks and 55.07 MiB/s for 512 KiB blocks.
- A temporary kernel benchmark mapped buffers once and then repeatedly submitted
  DMA copies.  It measured 20.92 MiB/s with 64 KiB transactions and 95.78 MiB/s
  with 512 KiB transactions.  The temporary benchmark hook was removed.  This
  shows that small DMA transactions are dominated by descriptor/IRQ overhead,
  while large transfers outperform cached software memcpy by about 1.74x.
- Normal dmatest (about 1.2 MiB/s as reported against the requested copy size)
  maps and synchronizes the full test buffers on every iteration, so it is an
  end-to-end DMA API result and must not be interpreted as raw AXI bandwidth.
